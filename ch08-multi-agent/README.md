# Ch08：Multi-Agent 与协议全景

> **Multi-Agent 的真正问题，从来不是“能不能多开几个 Agent”，而是系统是否值得为分工协作付出更高的通信、协调和一致性成本。**

走到这一章，单 Agent 的边界已经越来越明显了。

它会：

- 变得越来越胖
- 负责越来越多角色
- 同时承担规划、检索、执行、评估、整合

于是系统开始滑向一个老朋友：

> **God Object。**

## 8.1 从上一章说起：单 Agent 的宿命

如果一个 Agent 什么都做，它迟早会碰到三个问题：

- 提示词越来越长
- 工具集越来越乱
- 上下文越来越混杂

更严重的是，它还会同时承担互相冲突的目标：

- 既要做研究
- 又要写代码
- 又要审查结果
- 又要和外部系统协作

这就是为什么 Multi-Agent 会出现。

它不是为了“更酷”，而是为了把一个越来越像单体系统的 Agent，重新拆回有边界的协作结构。

---

## 8.2 Part A：多 Agent 编排模式

### 模式一：Orchestrator - Workers

这是最直觉的一种模式。

一个总控 Agent 负责：

- 理解总任务
- 拆分子任务
- 调度 Worker
- 汇总结果

Worker 则各自专注于具体职责。

这种模式适合：

- 任务结构相对清晰
- 角色边界较明确
- 需要一个中心做最终整合

### 模式二：Peer-to-Peer

在这种模式里，Agent 之间更像对等协作节点。

它们不一定有单一总控，  
而是根据需要彼此请求帮助、交换结果、推进任务。

这种模式更灵活，但也更难治理。

### 模式三：Hierarchical

分层模式更像组织管理结构。

高层 Agent 负责战略和分配，  
中层负责协调，  
底层负责执行。

它适合大任务，但也最容易把系统变复杂。

### 什么时候该选哪一种

Multi-Agent 没有万能答案。  
选择模式时最重要的问题不是“哪种最先进”，而是：

- 任务是不是天然可以拆分
- 各角色之间的信息损失大不大
- 协作成本会不会超过收益

这很像服务拆分问题。  
不是“能拆就拆”，而是“拆了之后整体是否更好”。

---

## 8.3 Part B：四层协议栈的全景

Multi-Agent 真正让问题升级的，不只是“多了几个 Agent”，  
而是系统突然开始需要一整套协议分层。

### Layer 1：Function Calling

这是原点。

模型先学会以结构化方式表达工具调用意图。  
没有这一层，后面所有能力都很难稳定生长。

### Layer 2：MCP

MCP 解决的是外部能力如何标准化接入。

它让 Agent 不必为每个工具、每个资源都手写私有适配。

### Layer 3：A2A

当一个 Agent 需要另一个 Agent 时，问题就不再只是“调工具”。

而是：

- 如何发现对方
- 如何描述任务
- 如何交换过程信息
- 如何接收最终产物

这就是 Agent-to-Agent 协议的意义。

### Layer 4：ACP

再往上一层，问题变成：

> 人类和 IDE 如何使用 Agent？

ACP 这类协议关心的是：

- Agent 怎么被 IDE 发现
- 会话怎么建立
- 消息怎么传
- 上下文怎么注入
- 结果怎么在界面中呈现

这时协议栈终于从“模型怎么用工具”，扩展到“人类怎么使用 Agent 系统”。

---

## 8.4 为什么这些协议不是竞争关系

很多人第一次看到 MCP、A2A、ACP，会问：

> 到底哪个会赢？

这个问题本身就有点偏。

更准确的理解是：

- FC 解决模型和工具表达的问题
- MCP 解决 Agent 和外部能力接入的问题
- A2A 解决 Agent 和 Agent 协作的问题
- ACP 解决人类工作环境和 Agent 交互的问题

也就是说，它们更像分层，而不是互斥替代。

这和网络协议栈、服务协议栈、IDE 协议栈的成熟过程非常相似。

---

## 8.5 事件聚焦：为什么 2026 会进入“协议全景”讨论

当行业开始大量讨论协议时，通常说明一件事：

> 生态已经不想继续忍受碎片化了。

到了 2025-2026，几个变化会汇合在一起：

- Agent 系统数量快速增长
- 不同厂商各自推出自己的接入方式
- 多 Agent 协作开始从实验走向产品
- coding agent 场景迫切需要 IDE 级标准接口

这时，协议问题就不再是附属话题，而开始成为主线。

从这个角度看，协议之争不是枝节。  
它本身就是基础设施成熟的一部分。

## 8.6 如果不写代码，怎么在脑海里理解 Multi-Agent

假设你要做一个 coding agent 系统，目标是修一个线上 bug。

一个合理的多 Agent 流程可能是：

1. 总控 Agent 接收任务
2. Research Agent 先搜索相关资料和历史 issue
3. Code Agent 基于这些信息形成修改方案
4. Review Agent 再对修改做审查
5. 总控 Agent 最后整合成可交付结果

如果把这个流程写成一段最小交互，会更接近真实系统里的样子：

```text
Coordinator -> Research Agent
"请找出 payment timeout 的历史 issue、相关提交和最佳实践。"

Research Agent -> Coordinator
Artifact:
- 找到 2 个历史 issue
- 最近一次类似问题由重试参数引起
- 相关提交位于 commit abc123

Coordinator -> Code Agent
"基于这些线索，给出修复方案，重点看 retry config。"

Code Agent -> Coordinator
Artifact:
- 建议回滚 timeout 配置
- 建议把连接池上限从 20 调到 50
- 给出修改 diff

Coordinator -> Review Agent
"请检查这个修复是否有副作用。"

Review Agent -> Coordinator
Artifact:
- 逻辑基本正确
- 但缺一个回归测试
```

这个例子能看出 Multi-Agent 的两个核心现实：

- 每个 Agent 只处理自己的一段职责
- 系统价值高度依赖中间结果能不能顺畅传递

这个例子说明了两件事：

第一，Multi-Agent 的价值在于**角色分化**。  
第二，Multi-Agent 的代价在于**信息必须跨角色传递**。

所以关键从来不是“多几个 Agent”，  
而是“拆分后的边界是否真的比单体更健康”。

## 8.7 深入剖析：几个反直觉的洞察

### 洞察 1：多 Agent 系统会重演分布式系统的老问题

一旦 Agent 之间开始通信，你就会重新遇到这些经典问题：

- 延迟
- 一致性
- 部分失败
- 重试
- 权限传播
- 上下文丢失

所以 Multi-Agent 的难点，从来不是“怎么起多个会话”。  
而是“怎么管理一个分布式协作系统”。

### 洞察 2：Prompt Injection 会在多 Agent 系统里传播

一旦一个 Agent 把污染过的输出交给下一个 Agent，风险就不再停留在本地。

这意味着：

- 输入隔离变重要
- 角色权限最小化变重要
- 中间结果的信任边界变重要

多 Agent 系统里的安全问题，往往比单 Agent 更像供应链风险。

### 洞察 3：协议会服从“引力法则”

生态不会长期容忍过多相互重叠、互不兼容的协议。

最终总会出现整合、分层、合并、角色重划。  
因为系统复杂度会逼着协议体系逐渐收敛。

### 洞察 4：少即是多

这可能是 Multi-Agent 最反直觉、也最实用的经验。

两个边界清楚的 Agent，往往比五个随意拆出来的 Agent 更好。

拆分只有在以下条件成立时才值得：

- 职责真的不同
- 中间信息不会大量损失
- 协作成本低于单体膨胀成本

否则，你只是把一个复杂问题拆成了多个更难协调的问题。

## 8.8 遗留痛点：从零件到产品

走到这里，你已经拥有了很多组件：

- Function Calling
- RAG
- MCP
- Agent
- Skill
- Memory
- Agentic RAG
- Multi-Agent
- 多层协议

但一个更大的问题也摆在面前：

> 就算零件都齐了，它还是不自动等于产品。

真正的产品级系统还需要：

- 权限门控
- 会话管理
- 后台任务
- 长任务续跑
- Feature Flags
- 子 Agent 管理
- 可运维的系统骨架

也就是说，系统现在缺的不是“再多一个能力”，  
而是一个把所有能力组织起来的操作层。
