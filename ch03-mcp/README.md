# Ch03：MCP — 工具与数据接入的标准化

> **MCP 的意义，不在于又多了一种工具接入方式，而在于它第一次尝试把“模型如何发现、读取、调用外部能力”这件事提升为一个开放协议。**

到了这一章，前两章的两条线已经开始汇合了。

- Ch01 解决的是：模型怎么表达工具调用意图
- Ch02 解决的是：模型怎么拿到外部知识

但当你真把系统做起来，很快会遇到同一个新问题：

> 工具接入、资源接入、提示接入，全都太碎了。

## 3.1 问题引入：两条支线终于撞到一起

Function Calling 让模型开始会“伸手”。  
RAG 让模型开始能“查资料”。

但工程上你很快会发现，这两件事的接入方式并不统一：

- 工具是工具的一套格式
- 资源是资源的一套方式
- prompt 模板又是另一套逻辑
- 不同模型厂商的调用接口还各不相同

这时候问题已经不是“模型会不会”，而是：

> **系统有没有一个统一的接入协议。**

如果没有，后果会非常熟悉：

- 每接一个数据源就要手写一层适配器
- 每换一个模型厂商就要改一轮集成代码
- 工具和资源难以复用
- 整个 AI 应用栈越来越像一堆胶水脚本

MCP 试图解决的，就是这个标准化问题。

---

## 3.2 历史背景：从 ChatGPT Plugins 到 MCP

### 2023 — ChatGPT Plugins：方向对了，但抽象层次不对

ChatGPT Plugins 已经证明：

> 模型和外部世界之间，确实需要一个标准接口层。

但它的问题也很明显：

- 更像产品机制，而不是底层协议
- 强依赖特定平台
- 对工程复用并不友好

它像一个重要的先驱，但不是最终答案。

### 2024.11 — Anthropic 发布 MCP

2024 年 11 月，Anthropic 正式发布 Model Context Protocol。

这一步的关键，不是“又出了一个新名词”，而是：

> **模型外部能力接入，第一次开始像真正的协议层演进。**

MCP 把问题重新定义成：

- 客户端如何发现能力
- 客户端如何读取资源
- 客户端如何调用工具
- 客户端如何复用提示模板

也就是把“能力接入”从零散实现推进到统一协议。

### 2025-2026 — 安全事件与生态爆发同时出现

MCP 一旦真的被广泛使用，两个现象几乎同时出现：

- 一边是生态快速增长
- 另一边是安全问题开始暴露，比如 tool poisoning 这类攻击

这反而说明它开始进入真正有工程影响力的阶段。  
只有变成基础设施，安全问题才会被系统性地看见。

## 3.3 核心概念：MCP 到底是什么

### 3.3.1 MCP 不是格式，而是协议

这是理解 MCP 最重要的一点。

MCP 不是一个“JSON 长什么样”的规范而已。  
它真正关心的是：

- 谁是 client
- 谁是 server
- 双方怎么发现能力
- 双方怎么交换上下文
- 双方怎么调用和返回结果

所以 MCP 的意义不只是“统一字段名”，而是：

> **让模型世界里的外部能力接入，第一次有了类似协议栈的抽象。**

### 3.3.2 MCP 的基本架构：Client / Server

MCP 可以很自然地理解成一个 Client-Server 模型。

大体上是：

- MCP Server 暴露能力
- MCP Client 发现并使用这些能力
- LLM 或 Agent 通过 Client 间接接触外部世界

这让模型不再需要为每个工具、每个资源都写一套私有集成逻辑。

### 3.3.3 三类能力：Tools / Resources / Prompts

MCP 的价值很大程度上来自于它不是只抽象了“工具”。

它至少把三类能力放进了同一个协议空间里：

**1. Tools**

Tools 是可执行的动作。  
比如搜索、查询、写入、执行、调用 API。

它对应的是“让模型能做事”。

**2. Resources**

Resources 是可读取的数据。  
比如文档、文件、数据库内容、配置、上下文对象。

它对应的是“让模型能拿到外部信息”。

**3. Prompts**

Prompts 是可复用的提示模板。  
这类能力经常被低估，但它其实是复用工作流、上下文注入、任务模板的重要基础。

三类能力放在一起的意义非常大，因为它说明 MCP 不再只把外部世界看成“工具集合”，而是开始把它看成一个更完整的能力平面。

### 3.3.4 传输层不是重点，协议层才是重点

MCP 可以跑在不同的传输方式之上，比如 `stdio` 或 HTTP。

但真正重要的不是传输细节，而是：

> **上层能力语义被统一了。**

这和很多基础设施协议的发展轨迹很像。  
大家最后记住的，不是底层用的是哪条线，而是“这套交互终于能被不同系统共同理解”。

---

## 3.4 FC vs MCP：它们不是竞争关系

很多人第一次看到 MCP，会误以为它是 Function Calling 的替代品。

其实不是。

两者解决的问题不在同一层：

| 层次 | Function Calling | MCP |
| --- | --- | --- |
| 解决的问题 | 模型如何表达调用意图 | 系统如何标准化暴露外部能力 |
| 关注点 | 函数名、参数、结构化输出 | 工具、资源、提示模板、发现机制 |
| 位置 | LLM 与宿主程序之间 | Agent / Client 与外部能力网络之间 |

所以更准确的理解是：

> Function Calling 是原点，MCP 是把这件事做成协议层。

前者解决“模型怎么开口说自己想调工具”。  
后者解决“系统怎么把能力以统一方式暴露出来”。

## 3.5 不写代码，也该在脑海里跑通一次 MCP

第一版里，这一章不提供代码。

但你完全可以在脑海里跑通一个最小场景：

假设你在做一个 coding agent，它需要：

- 搜索代码库
- 读取配置文件
- 复用一个“生成修复说明”的 prompt 模板

如果没有 MCP，你通常得：

- 手写一个搜索工具适配器
- 再写一套文件读取逻辑
- 再把 prompt 模板硬编码在应用里

如果有 MCP，这些能力都可以作为标准化暴露的能力存在。

从 Agent 的视角看，差别非常明显：

> 它面对的不再是一堆散乱的私有接口，而是一个统一的能力面。

如果把这个过程写成一段最小交互，MCP 的感觉会更具体一些。

假设一个 MCP Server 暴露了三类能力：

- Tool: `search_code`
- Resource: `repo://docs/architecture.md`
- Prompt: `fix_bug_summary`

那么一个 MCP Client 和它之间的交互，大致会像这样：

### 第一步：先发现这个 Server 有什么能力

```json
{
  "method": "tools/list"
}
```

返回：

```json
{
  "tools": [
    {
      "name": "search_code",
      "description": "在代码库中搜索函数名、错误信息和关键字"
    }
  ]
}
```

再查资源：

```json
{
  "method": "resources/list"
}
```

返回：

```json
{
  "resources": [
    {
      "uri": "repo://docs/architecture.md",
      "description": "系统架构说明文档"
    }
  ]
}
```

### 第二步：按标准方式读取或调用

读取资源：

```json
{
  "method": "resources/read",
  "params": {
    "uri": "repo://docs/architecture.md"
  }
}
```

调用工具：

```json
{
  "method": "tools/call",
  "params": {
    "name": "search_code",
    "arguments": {
      "query": "payment timeout"
    }
  }
}
```

返回：

```json
{
  "content": [
    {
      "type": "text",
      "text": "在 payment_service.py 中找到 3 处 timeout 相关逻辑"
    }
  ]
}
```

这个例子里最重要的不是字段名本身，而是：

> Client 不需要知道这个搜索工具背后是本地脚本、HTTP 服务还是别的实现。  
> 它只需要按同一套协议发现、读取和调用。

这就是 MCP 真正的产品价值。

## 3.6 安全隐患：为什么标准化也会放大风险

MCP 并不只带来好处。  
它也把风险集中暴露了出来。

其中一个很典型的问题就是 Tool Poisoning。

原理并不神秘：

- 如果模型会信任外部暴露出来的工具描述
- 而工具描述本身又可能被污染、误导或注入恶意内容
- 那么模型的决策过程就可能被间接操控

这很危险，因为攻击面已经不再只是“用户 prompt”。

它开始扩展到：

- 工具元数据
- 资源内容
- prompt 模板
- Agent 间传递的中间结果

所以 MCP 的成熟，不只是协议设计成熟，  
还意味着权限模型、信任边界、内容隔离和安全校验必须同步成熟。

## 3.7 深入剖析：几个反直觉的洞察

### 洞察 1：MCP 不是关于工具，而是关于协议

很多人第一次接触 MCP，会把它理解成“更方便接工具”。

但它真正重要的地方，是让 Agent 世界第一次开始有了“协议层思维”。

这和后端架构的成熟过程非常像。  
真正可扩展的系统，最终都会从“手写对接”走向“接口协议化”。

### 洞察 2：MCP 的成功，恰好建立在 Plugins 的失败经验上

Plugins 过于靠近具体产品形态。  
MCP 更靠近基础设施抽象。

这也是为什么它更容易被：

- 不同模型厂商
- 不同 Agent 系统
- 不同工具生态

共同接受。

### 洞察 3：Prompts 这类能力被严重低估了

很多人只关心 Tools。  
但从工程和产品视角看，Prompts 同样重要。

因为真正复杂的 Agent 系统，不只是需要“做什么”，  
还需要大量可复用的任务模板、上下文模板和协作模板。

这恰恰是 Prompt 作为协议能力被重新重视的原因。

### 洞察 4：对后端工程师来说，MCP 很像 gRPC / Service Mesh 的直觉

你完全可以把 MCP 理解成：

- 在 Agent 世界里，给外部能力建立统一契约
- 让调用方不必为每个能力单独适配
- 让生态围绕协议生长，而不是围绕单一平台生长

它不是一切的终点。  
但它是“系统开始成网”的前提。

## 3.8 遗留痛点：协议统一之后，新的问题是什么

到了这里，工具接入和数据接入终于开始变得更标准了。

但新的问题也立刻出现：

> 就算能力都已经接进来了，模型还是只会“单次调用”。

它依然不擅长：

- 自主规划多步任务
- 根据中间结果调整策略
- 在一串操作里持续迭代

换句话说，MCP 解决的是“能力怎么接入系统”。  
但它还没有解决“系统怎么自主完成任务”。
