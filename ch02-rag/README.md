# Ch02：RAG — 让模型获取外部知识

> **RAG 的本质，是在模型生成答案之前，先把相关外部知识找出来，再把它塞回模型上下文。它不是给模型“升级大脑”，而是给模型增加一个可控的查资料过程。**

如果说 Ch01 给了模型“手”，那 Ch02 解决的就是另一个同样根本的问题：

> 当模型根本不知道答案时，它怎么办？

## 2.1 问题引入：模型不知道训练之后的世界

Ch00 里我们已经看过，LLM 的知识被训练数据冻结住了。

这意味着它天然不知道：

- 训练之后的新事件
- 你公司的私有文档
- 某个团队刚刚更新的接口说明
- 某个用户的专属资料

这类问题不是靠“模型更聪明”就能解决的。  
问题不在推理能力，而在**缺少外部知识输入通道**。

最直接的工程解法也很朴素：

> **别让模型靠记忆硬答。先帮它把相关资料找出来，再让它基于这些资料回答。**

这就是 RAG。

## 2.2 历史背景：RAG 是怎么走到今天的

### 2020.05 — RAG 概念诞生

RAG 作为一个清晰概念，最早来自 2020 年的研究工作。

它提出的基本思路很简单：

- 先检索相关文档
- 再把检索结果交给生成模型
- 让回答建立在“可引用的外部上下文”上

这一步非常关键，因为它第一次把“生成”与“检索”明确接到了一起。

### 2023 — RAG 从概念走向主流实践

随着企业开始真正把 LLM 用于问答、客服、知识助手、文档系统，RAG 很快成为默认配置。

原因并不复杂：

- 企业真正关心的是私有知识
- 用户真正提的问题往往依赖最新资料
- 纯模型回答的幻觉风险太高

所以 RAG 很快从论文概念变成了 AI 应用里的基础设施。

### 2024 — Long Context、GraphRAG、复杂检索开始爆发

2024 年之后，RAG 不再只是“向量检索 + 拼接上下文”这么简单。

几个变化开始出现：

- 超长上下文模型出现，大家开始讨论 “Long Context 会不会杀死 RAG”
- GraphRAG 之类的方法开始处理关系型知识
- Query Rewriting、Reranking、Multi-hop 这类高级策略逐渐进入主流

这说明一件事：

> **RAG 不是一个固定技巧，而是一整套不断升级的检索增强系统。**

## 2.3 Part A：Naive RAG — 最朴素但最重要的基线

Naive RAG 的核心思想非常简单：

> **先检索，再生成。**

一个标准的 Naive RAG 通常包含六步：

1. 加载文档
2. 文档分块
3. 计算 embedding
4. 存入向量数据库
5. 根据用户问题检索相关片段
6. 把片段拼进上下文，让模型回答

如果用一句工程语言概括，它本质上就是：

> **在 LLM 前面接一层搜索系统。**

### 为什么这个基线重要

很多人一上来就想做 Advanced RAG。  
但如果没有 Naive RAG 这条基线，你根本不知道后面的复杂策略到底是在优化什么。

Naive RAG 的价值在于：

- 它提供了最基础的知识接入能力
- 它让很多私有知识问答场景第一次变得可行
- 它暴露了后面所有高级优化真正要解决的问题

### Naive RAG 最容易被低估的环节

看起来这条链路很简单，但里面有两个环节特别容易被低估：

**1. 分块（Chunking）**

分块不是一个机械步骤，而是整个系统精度的起点。

块太小：

- 上下文断裂
- 检索命中后信息不完整

块太大：

- 语义混杂
- 检索噪声上升
- 上下文成本变高

**2. 检索结果的质量**

RAG 的很多失败，并不是“模型回答错了”，而是“根本没找到正确资料”。

所以从工程视角看，RAG 的第一现场其实不在生成层，而在检索层。

## 2.4 Part B：Advanced RAG — 当 Naive RAG 不够用

一旦系统真正上线，你会很快发现 Naive RAG 有很多典型问题。

### 问题 1：用户问法和文档写法不一致

这时会出现 Query Rewriting。

模型先把用户问题改写成更适合检索的表达，再去搜索。  
本质上是在弥合“用户语言”和“文档语言”的差异。

### 问题 2：检索到了，但排序不好

这时会出现 Reranking。

第一轮检索负责粗召回，第二轮排序负责把真正最相关的片段排到前面。  
这和经典搜索系统里的“召回 + 排序”思路非常一致。

### 问题 3：片段命中了，但上下文不够

这时会出现 Parent Document Retrieval。

先用小块做检索，再回溯到更大的父文档或原始上下文。  
这样既保留小块检索的精确性，又避免回答时信息不完整。

### 问题 4：知识不是线性文本，而是关系网络

这时会出现 GraphRAG。

它试图回答的问题是：

> 当知识的关键不在单段文本，而在实体之间的关系时，普通向量检索还够不够？

GraphRAG 的意义，不在于它会取代一切，而在于它把 RAG 从“相似片段搜索”推进到“结构化知识导航”。

### 问题 5：问题太复杂，需要多跳推理

这时会出现 Multi-hop RAG。

系统不再一次检索完就回答，而是：

- 先检索一轮
- 根据结果再决定下一轮怎么检索
- 最后再把多轮检索结果综合起来

这说明检索开始从“静态流水线”走向“动态过程”。

### 问题 6：不同问题需要不同策略

这时会出现 Adaptive RAG。

不是每个问题都值得走最重的检索流程。  
有些问题用简单检索就够，有些问题需要重写、重排、多跳、图谱甚至外部搜索。

真正成熟的系统不会把所有问题都塞进同一条固定管线。  
它会根据问题类型动态选择检索策略。

## 2.5 如果不写代码，怎么在脑海里跑通一次 RAG

第一版里，这一章不提供代码。

因为在 RAG 里，最重要的不是先学某个向量库或框架，而是先建立一个稳定的系统直觉。

你可以在脑海里跑一个最简单的例子：

用户问：

> 我们公司的退款政策里，对年付企业版客户的退款窗口是多久？

这时系统内部发生的事情其实是：

1. 先把公司政策文档预处理成可检索片段
2. 根据用户问题做相似度检索
3. 找到最相关的政策片段
4. 把这些片段塞给模型
5. 要求模型严格基于检索结果回答

如果把这个过程再展开一点，系统内部的数据流大概会长这样。

假设知识库里原始文档被切成了很多 chunk，其中和退款最相关的两个是：

```text
Chunk A
Enterprise Annual Plan 退款规则：
年付企业版客户可在付款后 14 天内申请全额退款。

Chunk B
例外情况：
如果客户已经使用了专属实施服务，则退款窗口缩短为 7 天。
```

用户问题是：

> 我们公司的退款政策里，对年付企业版客户的退款窗口是多久？

程序做完检索后，真正发给模型的内容其实更像这样：

```text
System:
你是公司政策助手。请只根据给定 context 回答。
如果 context 不足，请明确说不知道。

User:
我们公司的退款政策里，对年付企业版客户的退款窗口是多久？

Context:
[1] Enterprise Annual Plan 退款规则：年付企业版客户可在付款后 14 天内申请全额退款。
[2] 例外情况：如果客户已经使用了专属实施服务，则退款窗口缩短为 7 天。
```

然后模型返回的回答才可能是：

```text
年付企业版客户默认可在付款后 14 天内申请全额退款。
如果已经使用了专属实施服务，则退款窗口缩短为 7 天。
```

这和“让模型凭空猜”最大的区别在于：

> **回答开始有依据了。**

从系统视角看，RAG 不是让模型“变聪明”，而是让回答过程多了一层证据来源。

## 2.6 深入剖析：几个反直觉的洞察

### 洞察 1：RAG 看起来简单，实则是一座冰山

表面上看，RAG 只是“检索一下再回答”。

但一旦真正落地，你会发现它背后其实是一整条复杂链路：

- 文档清洗
- 分块策略
- embedding 选择
- 向量存储
- 检索策略
- 排序策略
- prompt 组装
- 引用与归因

所以 RAG 从来不是一个小技巧。  
它更像一个缩小版搜索系统。

### 洞察 2：Long Context 不会自动杀死 RAG

超长上下文模型确实很强，但它并没有直接让 RAG 失效。

因为真正的问题不是“上下文能不能塞下”，而是：

- 你怎么知道该塞哪些内容
- 你怎么控制噪声
- 你怎么降低成本
- 你怎么让回答可追溯

Long Context 改变了边界条件，但没有取消检索的价值。

### 洞察 3：分块是最被低估的工程决策

很多团队花很多时间调模型，却对 chunking 采取默认配置。

但在 RAG 里，分块往往直接决定了：

- 召回质量
- 证据完整性
- 幻觉率
- 成本结构

它不是 preprocessing 的小细节，而是系统设计的一部分。

### 洞察 4：后端工程师看 RAG，本质上是在看搜索系统

如果你有搜索、索引、缓存、排序系统经验，你会发现 RAG 并不陌生。

它的新意在生成层。  
它的骨架其实仍然是检索系统工程。

这也是为什么后端架构师很适合做好 RAG：

> 你不是第一次在解决“如何从大量数据里稳定找到对的东西”。

## 2.7 遗留痛点：为什么 RAG 还不够

RAG 解决了“模型不知道”的问题，但它也很快暴露出新的限制。

### 痛点 A：检索策略开始爆炸

一旦进入 Advanced RAG，你会发现系统里很快长出越来越多策略：

- query rewriting
- reranking
- parent document retrieval
- graph retrieval
- multi-hop
- adaptive routing

这意味着检索不再是一个固定模块，而开始需要一层更高的编排能力。

### 痛点 B：工具接入和数据接入仍然是碎片化的

Ch01 里，Function Calling 已经让模型开始“会用工具”。  
Ch02 里，RAG 又让模型开始“能查资料”。

但这两条线的工程接口依然是碎的：

- 工具怎么定义
- 资源怎么暴露
- 提示模板怎么复用
- 不同厂商怎么兼容

这里已经隐约能看见一个新的需求：

> **需要一个标准化层，把工具接入和数据接入统一起来。**

这正是下一章要解决的问题。
