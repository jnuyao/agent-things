# Ch10：工程实战 — 把 Agent 推向生产

> **Agent 能不能真正上线，最后决定因素从来不是模型有多聪明，而是系统有没有足够强的安全边界、可观测性、成本控制、恢复能力和评估体系。**

如果说前面所有章节都在搭能力，  
这一章就是在回答最后那个最现实的问题：

> 当 Agent 不再只是 demo，而真的要面对用户和生产流量时，会发生什么？

## 10.1 问题引入：Demo Day 之后的噩梦

很多 Agent 系统在演示时都表现很好：

- 会调用工具
- 会查资料
- 会写代码
- 会跑多步流程

但一旦上线，真正的噩梦才开始：

- 一个危险工具被误调用
- 一轮检索拖慢整条链路
- token 成本突然失控
- Agent 卡在循环里不退出
- 结果看起来对，实际却不可用

也就是说，真正上线以后，问题会迅速从“能力展示”转向“工程治理”。

---

## 10.2 安全：从 Sandbox 到 Permission Model

### 为什么 Agent 安全比传统应用安全更难

传统系统通常行为比较固定。  
Agent 则会：

- 动态选择工具
- 动态生成参数
- 动态组合步骤
- 动态读写外部环境

所以它的风险面更大，也更难靠静态规则完全覆盖。

### Sandbox 是第一层防线

最基本的一步，是把 Agent 放进受控环境里。

也就是：

- 限制文件系统访问
- 限制网络访问
- 限制命令执行
- 限制资源消耗

这不是为了让系统“看起来专业”，  
而是因为没有隔离，很多错误会直接变成事故。

### Permission Model 是第二层防线

仅靠沙箱还不够。

系统还需要知道：

- 哪些工具默认可用
- 哪些工具需要用户确认
- 哪些工具只有特定角色可以使用
- 哪些工具永远不能暴露

Agent 世界里的权限模型，某种程度上就是传统 RBAC 在新运行时中的再发明。

### Prompt Injection 防护是第三层防线

Agent 会读外部内容、调用外部工具、接收中间结果。  
这意味着它很容易被污染输入影响。

Prompt Injection 不再只是聊天小把戏，  
而开始变成一个真实的系统安全问题。

所以安全边界必须逐层建立：

- 输入隔离
- 工具权限最小化
- 结果校验
- 高风险动作确认

## 10.3 可观测性：Agent 需要被看见

### 为什么传统 APM 不够

传统 APM 更擅长观察：

- 请求时间
- 错误率
- 资源消耗

但 Agent 系统还需要你看见：

- 它为什么做了这个决定
- 它调用了哪些工具
- 哪一步最耗 token
- 哪轮循环出了问题

也就是说，Agent 的可观测性不仅是“系统运行得怎样”，  
还包括“行为过程到底发生了什么”。

### Tracing 是核心能力

最自然的做法，是把 Agent 行为拆成一串可追踪的 span：

- think
- act
- observe
- retry
- summarize

这样你才能真正回答：

- 问题卡在哪一轮
- 成本花在哪一步
- 错误是模型问题还是工具问题

### Token 预算和工具成功率也必须纳入观测

很多 Agent 系统的真实瓶颈，不是 CPU，也不是内存，而是：

- token 消耗
- 工具调用成功率
- 外部依赖的延迟和失败

所以真正的可观测性，不只是 trace，更是：

- 成本观测
- 行为观测
- 工具观测

---

## 10.4 成本：没有成本控制，就没有产品

Agent 看起来很强，但也很容易变贵。

成本通常来自几块：

- 模型调用
- 长上下文
- 多轮循环
- 检索与外部工具
- 子 Agent 协作

### Prompt Caching 是最重要的 80/20 优化之一

很多系统第一眼看 Prompt Caching，会觉得这是小优化。  
实际上，它经常是最先见效、影响最大的降本手段。

只要系统里存在：

- 稳定的长系统提示
- 固定 Skill
- 可复用模板

Caching 就能显著改变成本结构。

### 上下文压缩是第二个核心杠杆

Agent 越能跑，越容易上下文膨胀。

如果没有压缩机制，成本和噪声会一起增长。  
所以系统必须学会：

- 保留什么
- 总结什么
- 丢弃什么

### 按需加载工具和上下文，是另一种成本纪律

不要默认把一切都暴露给 Agent。  
越少、越准、越按需，通常越便宜也越稳定。

---

## 10.5 错误恢复：Agent 的不确定性必须被驯服

### Agent 的错误和传统系统不完全一样

传统系统常见的是：

- 明确报错
- 明确超时
- 明确返回异常

Agent 系统则还会出现：

- 在错误路径上越走越远
- 卡在循环里不退出
- 工具失败后反复尝试错误策略
- 结果表面通顺但本质偏航

### 工具重试需要策略，而不是盲目重试

不是所有失败都该立刻重试。  
系统需要知道：

- 这是瞬时错误还是逻辑错误
- 要不要换参数
- 要不要换工具
- 要不要请求人工确认

### 卡死检测和自动恢复必须内建

一旦 Agent 进入长循环，没有卡死检测几乎是灾难。

系统需要能识别：

- 相同动作反复出现
- 长时间没有推进
- 上下文不断膨胀但目标没收敛

### Human-in-the-Loop 不是失败，而是成熟

很多团队把人工介入看成“系统不够自动化”。

其实恰恰相反。

真正成熟的系统知道：

- 什么时候该自动做
- 什么时候必须停下来问人

高风险动作、边界不清问题、信心不足时的人类介入，  
本来就应该是产品设计的一部分。

## 10.6 评估与测试：怎么知道 Agent 到底好不好

### 为什么 Agent 评估比传统系统难

同一个问题，Agent 可能给出多个都“看起来正确”的答案。

差异会出现在：

- 细节完整度
- 行为路径
- 风格
- 后续引导

所以 Agent 评估不只是判断“答对没答对”，  
还要判断过程和结果是否满足场景目标。

### RAG 检索质量需要单独评估

一个回答差，不一定是模型差。  
很可能是检索差。

所以系统必须区分：

- 检索失败
- 推理失败
- 工具失败
- 汇总失败

### 行为正确性同样要测

Agent 的评估不能只看文本输出。  
还要看它：

- 是否调用了正确工具
- 是否遵守权限
- 是否在该停的时候停
- 是否在该求助时求助

### Prompt 和策略变更需要回归测试

Agent 系统里，一个看似小小的 prompt 变化，  
都可能导致行为层面的系统性回归。

所以真正的测试对象，不只是函数输出，  
还包括：

- 策略
- prompt
- Skill
- 上下文装配

## 10.7 如果不写代码，怎么在脑海里理解“生产化”

你可以把生产化理解成五个问题：

1. **它会不会做危险的事？**  
   这是安全问题。
2. **出了问题我能不能看见？**  
   这是可观测性问题。
3. **它值不值得长期运行？**  
   这是成本问题。
4. **它失败后能不能恢复？**  
   这是错误恢复问题。
5. **我怎么证明它真的变好了？**  
   这是评估问题。

如果把这五个问题压缩进一次真实的生产回放，可能会像这样：

```text
09:00 用户请求：帮我查最近订单状态
09:00 Agent 调用 search_orders
09:00 Trace 记录：turn=1, tool=search_orders, tokens=2.1k

09:01 search_orders 超时
09:01 系统判断：这是工具失败，不是模型失败
09:01 自动重试一次，并记录 retry span

09:02 第二次仍失败
09:02 Permission Model 阻止 Agent 改用高风险数据库写入工具
09:02 系统触发 Human-in-the-Loop，请求人工确认是否切换到客服后台接口

09:03 人工确认
09:03 Agent 改用低风险只读工具 get_order_detail
09:03 返回结果，并生成最终回复

09:04 监控面板更新：
- 本次 trace 总耗时 4 分钟
- tool failure 2 次
- 总 token 消耗 5.8k
- 最终用户任务成功
```

这段回放里，其实同时包含了生产化的五个核心问题：

- 安全边界有没有生效
- trace 能不能看见全过程
- 重试和人工介入是否合理
- token 和失败成本有没有被记录
- 最终结果是否真正完成了用户目标

如果这五个问题没有答案，系统就算再会写 demo，也还不能算产品。

## 10.8 深入剖析：几个最终洞察

### 洞察 1：Agent 的可靠性主要来自工程约束，不来自模型本身

模型再强，也不能替代：

- 权限边界
- 运行时约束
- 可观测性
- 测试与回归

真正让系统活下来的，是工程，不是幻想。

### 洞察 2：Prompt Caching 经常是降本最有效的一击

很多人会优先优化模型切换、路由或小参数。  
但在真实系统里，Caching 往往是最先带来巨大收益的手段。

### 洞察 3：评估是最被低估的环节

没有评估，你其实不知道系统有没有进步。  
你只是在感受它“好像变好了”。

### 洞察 4：Agent 工程的目标正在从“让它做对”转向“让它不做错”

这是一种非常重要的成熟信号。

早期大家关注的是能力边界。  
真正上线后，系统最关心的往往是：

> 它能不能在出错时不造成更大伤害。

## 10.9 全书收束：从 Function Call 到 Agent OS

走到这里，整条线终于闭环了。

这本项目讲的从来不是 11 个并列名词。  
它讲的是一条问题驱动的演进链：

- 因为模型无法行动，所以有了 Function Calling
- 因为模型不知道外部知识，所以有了 RAG
- 因为接入太碎，所以有了 MCP
- 因为单次调用不够，所以有了 Agent
- 因为方法需要复用，所以有了 Skill
- 因为系统会失忆，所以有了 Memory
- 因为检索需要被接管，所以有了 Agentic RAG
- 因为单 Agent 会膨胀，所以有了 Multi-Agent
- 因为零件不等于产品，所以有了 Agent OS
- 因为系统必须活下来，所以最终一切都会回到生产工程

这也正是整个项目最想讲清楚的一件事：

> **AI 应用不是按概念进化的，而是按问题进化的。**

历史事件帮助我们理解这些问题是怎么被逼出来的。  
后端架构师的视角帮助我们判断这些解法为什么能成立。  
而真正的产品思维，则要求我们始终把视线放在最后那个问题上：

> 这个系统，真的能被用户长期使用吗？
