# Ch04：Agent — 自主规划与多步执行

> **Agent 的关键，不是“多调几个工具”，而是让系统进入一个持续的循环：观察、判断、行动、再观察，直到任务真正完成。**

到了这里，前面几章的积木已经基本齐了：

- 模型会表达调用意图
- 模型能接入外部知识
- 模型开始有标准化的能力接入方式

但系统仍然缺少一件真正关键的东西：

> **完成任务，而不是只完成一次调用。**

## 4.1 从“工具调用”到“任务完成”

把工具接进来，并不等于系统就变成了 Agent。

你很快会遇到一个熟悉的痛苦：

- 搜索要不要先做
- 结果不够时要不要再查一次
- 失败了之后要不要换策略
- 多个工具之间该按什么顺序串起来

如果这些流程都由工程师手写死，你得到的仍然只是一个“复杂脚本系统”。

真正的 Agent 出现于一个更强的目标：

> **让系统自己在多步过程中做局部决策，并持续推进任务。**

这就是从“会用工具”到“会完成任务”的分水岭。

---

## 4.2 历史背景：从研究范式到工程现实

Agent 这个词在 AI 世界里被用得很滥。  
但真正有工程价值的 Agent，并不是一个模糊概念，而是几条明确脉络汇合的结果。

### 第一条线：ReAct

ReAct 把“推理”和“行动”放进同一个循环里。

它的关键贡献不在于名字，而在于一个非常强的结构：

- 先思考
- 再行动
- 看结果
- 再继续思考

这给了后面大量 Agent 系统最基本的执行骨架。

### 第二条线：工具与协议成熟

如果没有前几章那些基础设施，Agent 只是纸上谈兵。

Agent 之所以在工程上开始可行，是因为：

- 工具调用开始稳定
- 外部知识接入开始成熟
- 协议标准开始出现

也就是说，Agent 不是凭空出现的新物种。  
它是前面所有能力积累到一定程度后的自然结果。

### 第三条线：大家逐渐意识到“手写编排”撑不住了

当任务稍微复杂一点，你就会发现：

- 静态流程图越来越长
- 分支越来越多
- 中间状态越来越难管理
- 失败恢复越来越难写

这时候你要么继续把系统变成一个巨大的状态机，  
要么承认一件事：

> **需要把一部分局部决策权交给模型。**

这就是 Agent 真正开始被工程系统吸收的时刻。

---

## 4.3 核心概念：Agent 的四梁八柱

### 4.3.1 ReAct：推理与行动的基本骨架

Agent 最常见的骨架就是 ReAct。

它的核心不是“想很多”，而是让系统形成一个闭环：

```text
观察问题
  ->
决定下一步
  ->
执行一个动作
  ->
读取结果
  ->
再次决定下一步
```

这和单次问答最大的区别在于：

> **系统开始对中间结果作出反应。**

### 4.3.2 Agent Loop：真正的核心只是一层循环

如果把所有包装都去掉，Agent 最核心的东西其实很少：

- 当前目标
- 当前上下文
- 可用工具
- 一个循环
- 一个停止条件

看起来很简单，但这层循环的意义巨大。

因为从这里开始，系统不再只是“回答一个问题”，而是在“推进一个过程”。

### 4.3.3 一个完整 Agent 至少需要这些组件

如果从工程角度拆开，一个可用的 Agent 通常至少需要：

- **规划能力**：决定下一步做什么
- **工具能力**：真的能操作外部世界
- **记忆 / 状态**：知道自己已经做过什么
- **反思 / 调整**：在结果不够好时换策略
- **停止机制**：避免无限循环

少了任何一项，系统都会迅速退化。

### 4.3.4 工具设计比很多人以为的重要得多

很多团队讨论 Agent 时，注意力全在模型。  
但实际工程里，工具设计经常比模型选择更决定成败。

一个好工具应该：

- 边界清楚
- 描述准确
- 输入输出稳定
- 权限最小化
- 结果易于后续步骤消费

一个差工具则会让 Agent 看起来“很聪明，但总干砸”。

### 4.3.5 从 Prompt Engineering 到 Context Engineering

Agent 起来之后，一个重要的暗线也越来越清楚：

大家逐渐不再只关心“怎么写 prompt”，而开始关心：

> **系统在每一轮到底应该让模型看到什么。**

这就是 Context Engineering 的核心。

因为 Agent 的成败，往往不只取决于它有没有工具，  
还取决于它在每一步是否拿到了正确、及时、干净的上下文。

## 4.4 如果不写代码，怎么在脑海里跑通一次 Agent

第一版里，这一章不提供代码。

但你完全可以在脑海里跑一个典型任务：

> 帮我定位最近一个线上告警的原因，并给出修复建议。

一个真正的 Agent 不会只做一次调用。  
它更可能经历这样的过程：

1. 先分析问题，判断需要哪些信息
2. 调用日志搜索工具
3. 发现信息不够，再去查监控或变更记录
4. 初步形成假设
5. 再调用代码搜索或文档工具验证
6. 最后整理成结论和建议

如果把这个过程再压成一段“最小运行日志”，Agent 的味道会更明显：

```text
Turn 1
Think: 先确认告警具体发生在哪个服务、什么时间段
Act: search_logs("error rate spike around 10:00")
Observe: payment-service 在 10:03 开始出现大量 timeout

Turn 2
Think: 需要确认同一时间是否有发布或配置变更
Act: get_recent_deploys("payment-service")
Observe: 09:58 刚发布了一个新的重试配置

Turn 3
Think: 需要验证 timeout 是否和重试参数有关
Act: search_code("payment retry timeout config")
Observe: 新配置把下游超时从 5s 提高到了 30s，导致连接堆积

Turn 4
Think: 证据基本够了，可以形成原因分析和修复建议
Final: 建议先回滚重试配置，再补充连接池与超时保护
```

这里最关键的不是它“想了很多轮”，而是：

> 每一轮的下一步，都是由上一轮观察结果驱动的。

注意，这里面真正重要的不是“工具很多”。

真正重要的是：

> **系统会根据中间结果调整下一步。**

这就是 Agent 和普通流水线最大的差别。

## 4.5 深入剖析：几个反直觉的洞察

### 洞察 1：Agent Loop 很像后端世界里的事件循环

它不是一个神秘新结构。  
它更像一个持续处理事件、更新状态、驱动下一步动作的循环系统。

如果你理解过事件循环、任务调度、状态机，你会很快对 Agent 形成直觉。

### 洞察 2：最好的建议往往是“先别急着用 Agent”

这是很多人第一次接触 Agent 时最不愿意听，但最应该记住的一点。

如果一个任务：

- 目标明确
- 步骤固定
- 分支很少
- 没有太多不确定性

那一条明确的工程流水线往往比 Agent 更稳定、更便宜、更可测。

Agent 真正适合的是那些：

- 路径不固定
- 中间信息不完整
- 需要边做边判断
- 很难手写全流程的任务

### 洞察 3：工具设计往往比模型选择更重要

很多所谓的“Agent 不好用”，最后根本不是模型不够强，  
而是工具边界混乱、描述糟糕、权限过大、结果不可消费。

Agent 系统里，工具设计几乎相当于“可执行的环境设计”。

### 洞察 4：Context Engineering 不是 Prompt Engineering 的升级版

它们不是一回事。

Prompt Engineering 更像在优化一句话怎么说。  
Context Engineering 则是在设计：

- 这一轮看到哪些信息
- 哪些历史该保留
- 哪些工具该暴露
- 哪些噪声必须屏蔽

这是一个系统级问题，不再是文案技巧问题。

### 洞察 5：一旦进入循环，记忆问题会立刻爆炸

Agent 能跑多步以后，一个新问题马上变得非常尖锐：

> 它怎么记住自己已经做过什么？

这不仅关系到跨会话记忆，  
也关系到当前任务内的状态保持、压缩、恢复与总结。

所以从这一章往下走，Memory 几乎是必然出现的下一块积木。

---

## 4.6 遗留痛点：从“能做”到“做得更好”

Agent 已经让系统开始具备自主性。  
但新的问题也随之出现：

- 它会重复做相似的工作
- 它缺少稳定的经验复用
- 它每次都像从零开始
- 它的好做法很难沉淀下来

换句话说，Agent 解决了“能不能做”。  
接下来要解决的是：

> **能不能把做对的方法固化下来，并复用给下一次。**

这正是下一章 Skill 的主题。

**Ch05：Skill — 可复用的能力封装。**
