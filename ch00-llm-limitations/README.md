# Ch00：被困在房间里的天才 — LLM 的能力边界

> **LLM 像一个被困在房间里的天才：它能生成流畅文本，却天然受困于知识截止、无法行动和幻觉，而后续所有 AI 应用工程，本质上都在试图打破这三道边界。**

## 引子

想象一个天才，被锁在一间没有窗户的房间里。

这个房间里有人类有史以来写过的几乎所有文字，书籍、论文、代码、对话记录、百科全书、法律条文、社交媒体帖子。这个天才把所有这些都读了，而且记住了其中的统计规律：哪些词经常出现在哪些词后面，哪些句式在哪些语境下最常见。

现在你可以通过门上的一道缝隙传纸条给他。你写一个问题，塞进去；他写一个回答，递出来。

他的回答经常令人惊艳，流畅、连贯，有时甚至展现出深刻的“洞察”。但有一个问题：

> **他看不到窗外。** 他不知道今天天气如何。他不知道股票涨了还是跌了。他不能帮你打电话、发邮件、查数据库。他甚至不能做确定性的数学计算，他只是在“猜”最可能的下一个字。

这就是大语言模型（LLM）的本质处境。

2022 年 11 月 30 日，OpenAI 发布了 ChatGPT。两个月内用户突破 1 亿，成为历史上增长最快的消费级应用。

所有人都在惊叹它的能力，但很少有人停下来问：**它不能做什么？这些“不能”是偶然的还是结构性的？**

这一章，我们就来回答这个问题。

---

## 历史背景：三个关键时刻

在讨论 LLM 的能力边界之前，我们需要知道它是怎么走到今天的。三个事件定义了这段历史：

### 2017.06 — Transformer：一切的起点

2017 年 6 月，Google 团队发表了论文 *"Attention Is All You Need"*，提出了 Transformer 架构。这篇论文至今被引用超过 15 万次（Google Scholar），是 21 世纪被引用最多的论文之一：[Attention Is All You Need](https://arxiv.org/abs/1706.03762)。

Transformer 的核心创新是**自注意力机制（Self-Attention）**。它让模型在处理一个词的时候，能同时“看到”输入中的所有其他词，并计算每个词对当前词的重要程度。在此之前，主流的 RNN/LSTM 架构只能按顺序逐词处理，序列越长效果越差。

Transformer 让并行处理成为可能，让模型能处理更长的上下文，让“暴力堆数据和参数”这条路走得通。GPT、BERT、Claude、Gemini，今天所有主流 LLM 都建立在这个架构之上。

### 2022.03 — InstructGPT / RLHF：让模型“听话”

拥有海量知识的模型不等于好用的模型。原始的 GPT-3 虽然“什么都知道”，但经常不理解用户到底想要什么。你问它一个问题，它可能会续写一段维基百科，而不是直接回答你。

2022 年 3 月，OpenAI 发表了 InstructGPT 论文，提出了**人类反馈强化学习（RLHF）**的三阶段训练范式：[Training language models to follow instructions with human feedback](https://arxiv.org/abs/2203.02155)：

| 阶段 | 做什么 | 目的 |
| --- | --- | --- |
| SFT（监督微调） | 人类标注员编写高质量的问答示范，模型在这些数据上微调 | 教模型“好的回答长什么样” |
| RM（奖励模型） | 模型生成多个回答，人类标注员排序，训练一个奖励模型 | 教另一个模型“怎么判断回答好不好” |
| PPO（强化学习优化） | 用奖励模型的评分作为反馈，用 PPO 算法优化原模型 | 让模型自我改进 |

最惊人的结论是：**经过 RLHF 训练的 13 亿参数模型，被人类评估者认为优于未经对齐的 1750 亿参数原始 GPT-3。** 参数量差了 100 倍，但小模型更好用。这说明“听话”比“聪明”更重要，至少对于用户体验而言。

RLHF 直接催生了 ChatGPT。

### 2022.11 — ChatGPT：LLM 进入大众视野

ChatGPT 的发布是 LLM 从学术走向大众的分水岭。5 天内用户突破 100 万，两个月突破 1 亿月活跃用户。TikTok 用了 9 个月，Instagram 用了 2.5 年才达到相同规模。

但如果你在那个月深度使用 ChatGPT，你会很快发现：**它什么都“知道”，但什么都做不了。**

- 你问它：“帮我查一下今天北京的天气。”  
  它会回答：“我无法访问实时数据，但根据北京 12 月的历史天气……”
- 你问它：“帮我算一下 4567 × 8901。”  
  它可能会自信地给出一个错误答案。
- 你让它推荐一本书。  
  它可能会“编造”一本不存在的书，配上看似合理的作者和出版社。

这不是 ChatGPT 的“问题”，而是 LLM 的**结构性特征**。要理解为什么，我们需要回到最根本的问题：LLM 到底是什么。

---

## 核心概念：text in → text out

### LLM 是一个自回归的下一个 token 预测器

剥去所有光环之后，LLM 的工作原理可以用一句话概括：

> **给定一个文本序列，预测最可能的下一个 token。**

整个生成过程是一个循环：

```text
输入 token 序列 -> 模型前向传播 -> 输出词汇表上的概率分布 -> 采样一个 token -> 追加到序列 -> 重复
```

用伪代码表示：

```python
def generate(prompt: str, max_tokens: int) -> str:
    tokens = tokenize(prompt)

    for _ in range(max_tokens):
        probabilities = model.forward(tokens)
        next_token = sample(probabilities, temperature=0.7)
        tokens.append(next_token)

        if next_token == EOS_TOKEN:
            break

    return detokenize(tokens)
```

这意味着三个关键事实。

**1. 模型不“理解”，它在做统计预测**

当你问“法国的首都是什么”，模型不是在一个知识库里查找“法国 -> 巴黎”的映射。它是在预测：给定“法国的首都是”这个 token 序列之后，下一个最可能出现的 token 是什么。因为训练数据中“法国的首都是巴黎”出现了无数次，所以“巴黎”的概率最高。

**2. 模型不“知道自己不知道”**

模型对每个位置都会输出一个概率分布。它没有一个内部信号表示“我不确定”，它只有概率值的高低。一个自信的错误答案和一个自信的正确答案，生成机制完全相同。

**3. 模型的“知识”就是训练数据的统计压缩**

LLM 不是数据库。它没有存储任何具体的事实。它存储的是参数（权重），这些参数编码了训练数据中的统计规律。这就像一个人读了一万本书后凭记忆回答问题。他会记住大致内容和常见搭配，但可能会张冠李戴、混淆细节。

### text in → text out 意味着什么

理解了这个本质之后，“text in → text out”这五个字就有了精确的含义：

- **text in**：模型的输入只有文本，准确说是 token 序列。它看不到屏幕，也听不到声音。多模态模型在输入侧扩展了这一点，但本质仍然是编码后的 token 序列。
- **text out**：模型的输出也只有文本。它不能点击按钮、调用 API、执行代码、访问数据库。它能做的唯一一件事，就是输出下一个 token。
- **没有中间状态**：模型没有持久记忆、没有目标、没有规划能力。每次被调用，它只看到当前上下文窗口中的 token，然后生成下一个 token。

## LLM 的三个结构性缺陷

理解了 “text in → text out” 之后，LLM 的三个核心缺陷就不再神秘了。它们不是偶发问题，而是架构层面的必然。

### 缺陷一：知识截止（Knowledge Cutoff）

**问题**

模型的知识冻结在训练数据的截止日期。

LLM 的“知识”来自训练数据。训练完成后，模型的参数就固定了。它不知道训练日期之后发生的任何事情：今天的新闻、最新的股价、刚发布的 API 文档、你公司内部的私有数据。

这不是信息更新不够快的问题，而是**模型根本没有“更新信息”的机制**。它不能上网、不能查数据库、不能访问文件系统。它就是那个被锁在房间里的天才，房间里的书永远不会多一本。

**真实案例**

你问 2024 年初的 ChatGPT “DeepSeek V3 的性能怎么样”，它会告诉你它不知道这个模型，因为 DeepSeek V3 在 2024 年 12 月才发布。或者更糟，它可能会根据 “DeepSeek” 这个名字编造一段看似合理的介绍。

### 缺陷二：无法行动（No Agency）

**问题**

模型只能输出文本，不能执行任何操作。

你让 ChatGPT “帮我发一封邮件给张三”，它会写出邮件的内容，但它没有任何方式真正把这封邮件发出去。它没有手，没有网络连接，没有执行环境。它能告诉你“你应该执行 `git push`”，但它自己执行不了。

更微妙的是，**模型也不能做确定性的计算**。当你问 “4567 × 8901 等于多少”，一个计算器会精确执行 `4567 × 8901 = 40649667`。但 LLM 不是在执行乘法运算，它是在预测 “4567 × 8901 =” 之后最可能出现的 token 序列。对于训练数据中高频出现的简单算式，它的预测恰好是正确的。但对于训练数据中没见过的大数乘法，它的预测就靠不住了。

**后端工程师的类比**

这就像一个只有 `stdout` 的进程。它能打印信息，但不能写文件、不能发 HTTP 请求、不能操作数据库。一个只有 `stdout` 的进程能做的事情非常有限。

### 缺陷三：幻觉（Hallucination）

**问题**

模型会自信地生成不存在的事实。

幻觉是前两个缺陷的自然延伸。因为模型不能查证事实，又不知道知识边界在哪里，它的生成机制会在没有足够信息时“创造”看似合理的内容。

幻觉的三种典型形态：

| 类型 | 表现 | 示例 |
| --- | --- | --- |
| 编造事实 | 生成不存在的论文、书籍、事件 | 编造一篇看似真实的 arXiv 论文，有标题、作者、摘要，但完全不存在 |
| 篡改细节 | 真实事件的关键细节出错 | 把一个公司的创始人和另一个公司搞混 |
| 接受错误前提 | 不质疑问题中的错误假设 | 被问“瑞士在二战中加入同盟国后做了什么”，直接编造答案，而非纠正“瑞士是中立国” |

**真实案例**

2023 年 6 月，纽约南区联邦法院的 *Mata v. Avianca* 案中，律师使用 ChatGPT 进行法律研究，ChatGPT 编造了至少 6 个完全不存在的案例引用。有虚假的案件名、虚假的判例编号、虚假的法官意见。律师未经核实就提交给了法院，最终被罚款 5000 美元。这成为 LLM 幻觉在专业场景中造成实际损害的标志性事件。

**为什么幻觉是结构性的**

幻觉不是一个可以通过“更好的训练”完全消除的 bug。它是自回归生成的固有特性。模型被训练为“永远生成下一个 token”，而不是“在不确定时停止”。只要模型无法访问外部事实来源进行校验，幻觉就是不可避免的。

---

## 三个缺陷的统一视角

这三个缺陷并非独立存在，它们有一个共同的根源：

> **LLM 是一个封闭系统。** 它被困在 “text in → text out” 的循环里，与外部世界没有任何连接。

| 缺陷 | 本质 | 需要什么来弥补 | 后续章节 |
| --- | --- | --- | --- |
| 知识截止 | 没有信息输入通道 | 接入外部数据源 | Ch2 RAG |
| 无法行动 | 没有执行输出通道 | 接入外部工具 | Ch1 Function Calling |
| 幻觉 | 没有事实校验通道 | 接入检索与验证机制 | Ch2 RAG + Ch4 Agent |

这就是整个项目接下来要解决的问题。从 Ch1 到 Ch10，每一章都在打破“封闭系统”的某一面墙。

## 如何自己验证这些缺陷

第一版里，这一章不提供配套代码。

原因很简单：这一章最重要的不是“跑一个 demo”，而是先把问题看清楚。你完全可以用任何公开可用的聊天模型，自己做四组观察：

### 观察 1：实时信息

你可以向模型提问：

- 今天的天气
- 今天的股价
- 今天的新闻

预期结果：模型要么承认不知道，要么编造一个看似合理的答案。无论哪种，它都无法稳定给出准确的实时信息。

### 观察 2：精确计算

你可以让模型做从简单到复杂的数学计算，例如：

- `17 × 24`
- `4567 × 8901`
- 多步骤商业计算

预期结果：简单算式可能正确，但复杂算式和多步骤计算会明显暴露不稳定性。

### 观察 3：幻觉测试

用三个“陷阱问题”测试模型是否会编造不存在的内容：

- 要求介绍一篇不存在的论文
- 要求总结一本不存在的书
- 在问题中嵌入一个错误前提，看模型是否会纠正

### 观察 4：需要行动的问题

给模型一个明显需要连接外部世界的任务，例如：

- 发一封邮件
- 查询数据库
- 检查某个线上指标

预期结果：模型可以把过程描述得很清楚，但它自己并不能真正完成动作。

这四组观察足以帮助你建立一个非常稳的直觉：

> 模型的问题，不是“不够聪明”，而是“没有通向世界的通道”。

## 深入剖析：几个反直觉的认知

### 幻觉不是“模型不够好”，而是架构必然

很多人以为幻觉会随着模型变大、数据变多而消失。但从 GPT-3 到 GPT-4 到 GPT-4o 到 GPT-5，幻觉率在降低，却从未消除。因为问题的根源不在模型能力，而在架构设计。一个被训练为“永远输出下一个 token”的系统，在缺乏外部事实源时，必然会在自信和准确之间产生偏差。

### “知道很多”和“能做事”之间有一道鸿沟

ChatGPT 可以写出完美的 Python 代码来查询天气 API，但它自己无法执行那段代码。它可以给你一份详细的 SQL 查询，但它无法连接你的数据库去运行它。它像一个精通一切理论但双手被绑住的专家。

### 上下文窗口不是“记忆”

模型的上下文窗口，从 4K 到 100 万 token，很容易被误解为“记忆”。但上下文窗口是一次性的。每次 API 调用都是一个新的独立计算。上一次对话说了什么，模型完全不知道，除非你把它重新塞进上下文。这意味着每一轮对话，模型都在“失忆”后重新开始。

## 遗留痛点：下一章要解决什么

本章建立了一个清晰的认知框架：LLM 是一个被困在 “text in → text out” 循环中的封闭系统。它有三个结构性缺陷：知识截止、无法行动、幻觉。

这三个缺陷中，**“无法行动”是最紧迫的**。因为如果模型能调用外部工具，那么知识截止和幻觉的问题也可以部分解决。比如通过调用搜索引擎获取实时信息，通过调用计算器做精确计算，通过调用数据库校验事实。

所以下一章的问题是：

> **如何让一个只能输出文本的模型，学会“使用工具”？**

2023 年 6 月 13 日，OpenAI 给出了第一个答案：Function Calling。
