# Ch06：Memory — 给 Agent 装上长期记忆

> **Memory 的价值，不是让 Agent 看起来更像人，而是让它终于摆脱“每次调用都像失忆后重新开始”的状态。**

到了这里，Agent 已经会规划，也开始有可复用的 Skill。

但只要你真的连续使用它几次，一个新的痛苦就会立刻暴露出来：

> 它不记得你。  
> 它不记得项目。  
> 它甚至不记得自己上一次做到了哪里。

## 6.1 问题引入：一个“失忆”的 Agent 有多痛苦

Memory 之所以重要，不是因为它听起来高级。  
而是因为没有它，Agent 的很多体验根本无法成立。

### 场景 1：用户偏好反复重说

今天你告诉它：

- 你喜欢 Markdown
- 你讨厌长篇废话
- 你希望优先看结论

过几天再来，它又像第一次见你一样。

### 场景 2：长任务一做就断片

一个任务稍微长一点，例如分析几十个 API endpoint、跑多轮诊断、跨多个文件修复系统问题，当前上下文就会开始撑不住。

于是你会发现：

- 前面做过的分析丢了
- 中间决策无法复用
- 到最后只能基于最近一段上下文继续往下走

### 场景 3：跨 Session 完全断裂

周一聊完一个需求，周二再来继续，系统却像第一次接触这个项目。

这说明什么？

> 上下文窗口不是记忆。  
> 它只是一次调用里的临时工作台。

---

## 6.2 历史背景：从“无状态”到“有记忆”

早期 LLM 应用几乎都是无状态的。

每次调用都像：

- 给一段输入
- 拿一段输出
- 然后结束

但一旦系统开始变成 Agent，问题就变了。

### 第一阶段：大家先用最原始的方法补记忆

最早、最朴素的做法非常简单：

- 把关键信息写进文件
- 或者在下一轮重新塞进 prompt

这虽然粗糙，但已经说明需求真实存在：

> 系统必须能跨轮次保存一些信息。

### 第二阶段：向量化记忆开始出现

随着 embedding 和检索能力成熟，很多系统开始把“记忆”做成可检索对象。

这样系统就不必保存全部历史，  
而可以在当前任务里按需召回最相关的过去信息。

### 第三阶段：结构化记忆和分层记忆开始形成

再往后，大家开始意识到：

并不是所有记忆都应该被同一种方式存储。

有些适合写成显式规则。  
有些适合语义检索。  
有些适合结构化存储。  
有些则应该被主动遗忘。

这时 Memory 才真正开始从“补丁”变成“系统层能力”。

---

## 6.3 核心概念：Memory 到底是什么

### 6.3.1 记忆不是一块东西，而是一套层次

理解 Memory 最好的方式，是把它看成一种分层存储体系。

有些信息像寄存器：

- 当前一步正在做什么
- 这轮推理刚拿到什么结果

有些像内存：

- 当前任务已经形成的中间状态
- 这次会话内的关键上下文

有些像长期存储：

- 用户偏好
- 项目背景
- 长期决策
- 历史经验

一旦这样理解，你就会明白：

> 记忆系统的设计，不是“存不存”，而是“什么该存在哪一层”。

### 6.3.2 三种最常见的工程实现

**方案 1：文件式记忆**

最简单也最直接。  
比如 `MEMORY.md` 之类的文件，显式记录用户偏好、项目背景、关键决策。

优点：

- 简单
- 可读
- 可人工编辑
- 很适合早期系统

缺点：

- 结构不够强
- 检索不够灵活
- 容易膨胀

**方案 2：向量化记忆**

把历史片段 embedding 化，需要时做语义检索。

优点：

- 适合模糊召回
- 适合规模较大的历史内容

缺点：

- 精确控制较弱
- 容易召回看起来相关但其实不该用的内容

**方案 3：结构化记忆**

把记忆按实体、关系、类型或状态显式存储，比如数据库、键值对、图结构等。

优点：

- 可控性高
- 适合精确读写
- 易于约束和治理

缺点：

- 设计成本高
- 演进难度更大

真正成熟的系统，往往不是三选一，而是混合使用。

### 6.3.3 Memory × Skill：Agent 的“人格”开始出现

Skill 解决的是“怎么做”。  
Memory 解决的是“为谁做、基于什么背景做、延续什么历史做”。

一旦 Skill 和 Memory 结合，系统的行为就开始稳定起来：

- 同样的方法被复用
- 同样的偏好被记住
- 同样的项目背景被延续

这时候 Agent 才开始真正表现出持续性。

### 6.3.4 记忆不只是写入，也包括压缩与遗忘

很多人第一次设计 Memory，只关注“怎么存”。

但真正困难的问题往往是：

- 哪些该保留
- 哪些该压缩
- 哪些该丢弃

记忆系统一旦不控制膨胀，很快就会变成噪声仓库。  
而一个不能忘记的系统，最终也无法高效工作。

## 6.4 如果不写代码，怎么在脑海里理解 Memory

假设你连续三周都在和同一个 coding agent 协作。

你希望它逐渐记住这些事情：

- 你偏爱简洁回答
- 这个仓库是教育项目，不是产品代码库
- `ch00` 和 `ch01` 已经改成“长文 README”结构
- 某个命名方案已经被你否掉

如果它没有记忆，你每次都得重说。  
如果它有记忆，它就应该在新的任务里自动带上这些背景。

如果把这件事写成一个非常具体的记忆快照，可能会像这样：

```text
User Preferences
- 回答优先给结论
- 默认用中文
- 不喜欢无意义的鼓励语气

Project Context
- 仓库是 study_agent_history
- 各章节正在整理成单文件 README

Key Decisions
- ch00 不保留 lab
- ch01 需要补充真实交互示意
```

然后下一次你只说一句：

> 继续处理 ch02。

系统如果真的有记忆，它在进入这一轮时就应该自动知道：

- 你希望答案简洁
- 当前项目是教育内容仓库
- 前面章节已经采用什么结构

这就是 Memory 在产品层真正创造出来的连续性。

从这个例子里，你会发现 Memory 真正带来的不是“智能感”，而是：

> **连续性。**

这对产品体验比很多人想象的重要得多。

## 6.5 深入剖析：几个反直觉的洞察

### 洞察 1：记忆系统本质上是 Agent 的“有状态服务”

传统 LLM 调用是无状态函数。  
Memory 则把 Agent 推向了一个有状态系统。

一旦进入这个世界，你要面对的问题就会越来越像后端服务设计：

- 状态写在哪里
- 怎么读
- 怎么更新
- 怎么压缩
- 怎么恢复
- 怎么避免脏数据污染

### 洞察 2：`MEMORY.md` 这种“低技术”方案经常赢

这很反直觉，但在很多真实系统里，简单文件方案之所以有效，恰恰因为它：

- 足够透明
- 足够可控
- 足够容易人工介入

它未必最优雅，却经常最先可用。

### 洞察 3：遗忘和记忆同样重要

真正好的记忆系统，不是记得越多越好。  
而是知道：

- 哪些该长期保留
- 哪些该浓缩
- 哪些该让它自然消失

这和缓存、日志、归档系统的治理逻辑非常像。

### 洞察 4：Context Window 不是“内存”，更像“工作台”

工作台可以放当前正在处理的材料。  
但工作台不是仓库，更不是档案馆。

这就是为什么“大上下文”可以缓解问题，却不能替代长期记忆系统。

### 洞察 5：什么时候形成记忆，和形成什么记忆一样重要

如果过早写入，系统会把大量噪声当成长期事实。  
如果过晚写入，又可能错过真正重要的决策和偏好。

所以记忆系统真正困难的部分，并不只是存储，而是**记忆形成时机**。

---

## 6.6 遗留痛点：有了记忆，检索却还是静态的

到了这里，Agent 已经：

- 会规划
- 会复用 Skill
- 会保留长期记忆

但另一个问题开始变得明显：

> 当前很多 RAG 仍然像一条静态流水线。

它不会：

- 根据问题动态决定检索策略
- 根据中间结果继续追问
- 在多源知识之间自主切换
- 自己判断检索质量够不够

也就是说，Memory 让 Agent 记得住过去，  
但检索过程本身还没有真正被 Agent 接管。

这就是下一章的重点：

**Ch07：Agentic RAG — 当 Agent 接管检索。**
