# Ch07：Agentic RAG — 当 Agent 接管检索

> **Agentic RAG 的关键变化，不是“检索更复杂了”，而是检索本身终于从一条静态流水线，变成了 Agent 可以自主规划、反思和调整的能力。**

走到这一章，前面的几块积木终于开始真正组合起来：

- 有了工具
- 有了外部知识
- 有了协议
- 有了自主执行
- 有了 Skill
- 有了 Memory

这时候，一个自然的问题会出现：

> 如果 Agent 已经这么会规划了，为什么检索还要被固定在一条死管线上？

## 7.1 从 Memory 到 Agentic RAG：静态检索的瓶颈

传统 RAG 的典型流程是：

- 收到问题
- 检索一次
- 把结果拼给模型
- 直接回答

这在很多场景里足够用了。  
但一旦问题变复杂，瓶颈很快就会暴露。

### 瓶颈 1：多源检索能力不足

现实里的复杂问题往往同时涉及：

- 文档库
- 搜索引擎
- 图谱
- 数据库
- 历史记忆

静态 RAG 往往只能固定走一条路径。

### 瓶颈 2：不会追问，不会继续查

很多问题不是一次检索就够。

第一次结果可能只是暴露了新的关键词、新的实体、新的线索。  
真正好的系统应该会顺着这些线索继续查下去。

### 瓶颈 3：不会判断自己检索得好不好

传统 RAG 常见的默认假设是：

> 检索完了，就拿这些结果回答。

但真正的问题是：

- 这些结果够不够
- 相关性高不高
- 是否覆盖了关键点
- 是否需要重试

一旦系统不会自评，它就很容易在“检索其实没做好”的情况下硬答。

---

## 7.2 从 Naive RAG 到 Agentic RAG：一条自然演进线

### Ch02 的 Naive / Advanced RAG 已经埋下伏笔

回看 Ch02，你会发现很多 Advanced RAG 方法已经在暗示一件事：

- query rewriting
- reranking
- multi-hop
- adaptive routing

这些东西本质上都在说明：

> 检索不再只是一个固定步骤，而开始变成一个可决策过程。

### 2023-2024：Self-RAG、CRAG 等工作进一步推动了这个方向

这一时期出现的很多方法，核心都在往同一个方向推进：

- 模型自己判断要不要检索
- 模型自己判断结果够不够好
- 模型自己决定要不要修正策略

这正是 Agentic RAG 的雏形。

### Agentic RAG 的形成

当你把前几章所有能力放在一起，Agentic RAG 几乎是必然出现的：

- RAG 提供知识接入
- Agent 提供循环和规划
- Memory 提供历史与状态
- Skill 提供可复用检索方法

于是检索开始不再是一条固定管线，  
而变成 Agent 手里的一个能力模块。

## 7.3 核心概念：Agent 如何驾驭检索

### 7.3.1 最好的类比：开卷答题 vs 做研究

传统 RAG 很像开卷答题。

你先去翻几页资料，  
然后立刻回到桌上作答。

Agentic RAG 更像研究员做课题。

它会：

- 先制定检索思路
- 查第一轮资料
- 发现新线索
- 再继续搜索
- 对结果质量做判断
- 最后综合多轮信息形成答案

这就是两者最本质的差异。

### 7.3.2 四种核心模式

**模式一：动态检索策略选择**

不是所有问题都该走同一条检索路径。  
Agent 应该根据问题类型决定：

- 查向量库
- 查图谱
- 查搜索引擎
- 查内部记忆

**模式二：迭代检索 - 推理循环**

一次检索不够，就继续下一轮。  
每轮检索结果都可以影响下一轮策略。

**模式三：多工具协调**

一个问题可能需要多个能力一起工作。  
比如先搜索，再查数据库，再读文档，再回看历史记忆。

**模式四：自主评估检索质量**

Agent 不再假设“检索完就能答”。  
它要学会判断：

- 信息是否足够
- 是否存在冲突
- 是否需要补充证据

这让系统第一次开始真正具备“知道自己还没查够”的能力。

### 7.3.3 从单 Agent 到多 Agent 的影子已经出现

当检索越来越复杂时，另一个变化也开始出现：

有些系统会把：

- 查询规划
- 资料检索
- 结果评估
- 答案综合

拆给不同角色完成。

这说明 Agentic RAG 其实已经在悄悄逼近多 Agent。

## 7.4 如果不写代码，怎么在脑海里跑通一次 Agentic RAG

你可以想一个典型问题：

> “对比最近两年主流 Agent 协议的发展脉络，并说明它们对 coding agent 的意义。”

如果是传统 RAG，系统可能只是：

- 检索几段文档
- 然后直接回答

如果是 Agentic RAG，它更可能这样做：

1. 先判断问题涉及时间线、协议对比和应用意义
2. 第一轮查时间线资料
3. 第二轮查协议定义与差异
4. 第三轮查 coding agent 场景中的应用
5. 评估现有资料是否足够
6. 如果还不够，再补一轮检索
7. 最后再形成综合回答

如果把这个过程写成一段“最小检索日志”，你会更容易看出它为什么已经不是普通 RAG：

```text
Round 1
Plan: 先确认 2024-2026 的主要协议时间线
Retrieve: MCP 发布时间、A2A 发展、ACP 进入 coding agent 讨论
Judge: 时间线有了，但“对 coding agent 的意义”还不够

Round 2
Plan: 补查这些协议分别解决什么问题
Retrieve: MCP 解决外部能力接入，A2A 解决 Agent 间协作，ACP 解决 IDE / Agent 交互
Judge: 结构更清晰了，但还缺应用层例子

Round 3
Plan: 搜 coding agent 场景中的真实使用链路
Retrieve: IDE -> ACP, Agent -> MCP, Agent -> A2A, 内部 LLM -> FC
Judge: 信息足够，可以开始回答
```

这时候系统做的已经不是：

> “搜一次，然后答。”

而是：

> “边搜边判断，直到证据足够再答。”

这里真正重要的不是“查得更多”。  
而是：

> **检索开始被任务目标驱动，而不是被一条固定模板驱动。**

## 7.5 深入剖析：几个反直觉的洞察

### 洞察 1：Agentic RAG 是前六章的一次“集成测试”

走到这里，你会发现它几乎把前面的能力全都拉进来了：

- Ch01 的工具调用
- Ch02 的检索增强
- Ch03 的标准化接入
- Ch04 的多步循环
- Ch05 的可复用策略
- Ch06 的长期状态

所以 Agentic RAG 不只是一个子话题。  
它更像前半本书所有能力的一次合流。

### 洞察 2：检索应该是 Skill，而不是死管线

一旦这样理解，很多事情都会突然变清楚：

- 检索策略可以复用
- 检索流程可以被替换
- 检索能力可以随任务动态组合

这比“写死一条 pipeline”灵活得多。

### 洞察 3：过度检索同样有代价

Agentic RAG 并不意味着“查得越多越好”。

检索越多，代价就越高：

- latency 上升
- token 成本上升
- 噪声上升
- 系统更容易迟迟不停止

所以一个成熟系统必须有停止条件。  
这和递归需要 base case 是同一个道理。

### 洞察 4：控制权正在从静态配置转移给 Agent

传统 RAG 的控制权在工程师手里。  
Agentic RAG 则把一部分检索控制权交还给 Agent。

这会带来更强的灵活性，也会带来更大的系统复杂度。

### 洞察 5：Agentic RAG 很快会逼出单 Agent 的上限

一旦任务涉及：

- 多源知识
- 多轮检索
- 质量评估
- 答案合成

单个 Agent 很容易开始膨胀成 God Object。

这正是下一章要面对的问题。

## 7.6 遗留痛点：单 Agent 的天花板

Agentic RAG 让检索变得更聪明了。  
但也让系统变得更重了。

一个 Agent 既要：

- 规划检索
- 调用工具
- 评估质量
- 整合结果
- 维持状态

很快就会变成一个庞大而脆弱的单体。

于是下一个问题自然出现：

> 要不要把这些职责拆开，让多个 Agent 分工协作？

这正是下一章的主题：

**Ch08：Multi-Agent 与协议全景。**
