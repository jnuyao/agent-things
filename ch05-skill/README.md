# Ch05：Skill — 可复用的能力封装

> **Skill 的本质，是把“怎么把一件事做好”的经验，序列化成一个可以被 Agent 重复加载、重复执行、重复演化的能力单元。**

到了这一步，Agent 已经会：

- 规划
- 调工具
- 多步执行

但一个新问题马上就冒出来了：

> 每次做相似任务，难道都要从头再来一遍吗？

## 5.1 问题引入：当 Agent 陷入“复制粘贴”的泥潭

如果没有 Skill，很多 Agent 系统很快会滑向一种非常熟悉的状态：

- 同类任务反复写同样的指令
- 同样的流程在不同地方复制粘贴
- 稍微改一点就出现行为漂移
- 一次优化很难复用到下一次

这和早期软件工程里没有抽象、没有模块化、没有封装时的状态几乎一模一样。

所以 Skill 出现的真正原因，不是“让 Agent 看起来更强”。  
而是：

> **把经验沉淀下来，让它能复用。**

从工程语言看，Skill 解决的是经验复用和行为一致性问题。

---

## 5.2 历史背景：为什么 Skill 生态会崛起

### 第一条线：System Prompt 泄露让大家看见了“行为其实可以打包”

随着越来越多系统 prompt 被公开，人们开始逐渐意识到一件事：

很多所谓“Agent 的能力”，并不是神秘黑盒。  
它往往就是：

- 一组明确指令
- 一些辅助脚本
- 一批资源与模板
- 一套约束和输出格式

也就是说，经验本身是可以被封装的。

### 第二条线：Agent 实践逼着大家走向能力模块化

当 Agent 真正进入工程系统后，团队很快会遇到这些问题：

- 代码审查怎么复用
- 部署流程怎么复用
- Git 工作流怎么复用
- 某个团队独有的最佳实践怎么复用

如果没有 Skill，唯一办法就是把一切重新塞回系统 prompt。  
但这很快会变得不可维护。

### 第三条线：开放生态开始形成

一旦“能力可封装”的共识形成，生态自然会往前走：

- 有人开始整理可复用技能包
- 有人开始做 Skill 发现与加载
- 有人开始讨论 Skill 的元数据与依赖

这说明 Skill 不再只是某个团队内部技巧，  
而开始变成 Agent 系统中的正式构件。

---

## 5.3 核心概念：Skill 到底是什么

### 5.3.1 Skill 不是 Tool

这是最重要的区分。

Tool 解决的是：

> 系统能不能执行一个动作。

Skill 解决的是：

> 系统应该怎样更稳定、更专业地完成一类任务。

Tool 更像一个原子操作。  
Skill 更像一套经过封装的做事方法。

### 5.3.2 一个 Skill 通常由三部分组成

一个成熟的 Skill，往往至少包含三类东西：

**1. 指令（Instructions）**

告诉 Agent：

- 目标是什么
- 应按什么步骤执行
- 哪些规则不能违反
- 输出应该长什么样

**2. 脚本（Scripts）**

把一部分重复性、确定性的工作下沉成程序。  
让 Agent 不必每次都靠自然语言临时发挥。

**3. 资源（Resources）**

比如配置、模板、规则集、示例、知识文件。  
它们让 Skill 不只是“一段话”，而是一个完整能力包。

如果用一个最小的 `code-review` Skill 举例，它的结构可能会像这样：

```text
code-review/
  skill.yaml
  instructions.md
  scripts/
    run_lint.sh
  resources/
    review_checklist.md
```

其中：

- `skill.yaml` 负责描述这个 Skill 是做什么的
- `instructions.md` 规定执行步骤和输出格式
- `scripts/` 放确定性的辅助动作
- `resources/` 放审查清单、规范和参考资料

也就是说，Skill 从一开始就不是“多写一点 prompt”。  
它更像一个可以被加载的能力目录。

### 5.3.3 Skill 和普通 Prompt 的本质区别

普通 Prompt 更像一次性交代。  
Skill 更像一个可复用模块。

它的关键价值在于：

- 可以被发现
- 可以被加载
- 可以被复用
- 可以被版本化
- 可以被社区共享

从后端工程师视角看，Skill 有点像：

- 一个带约束的工作流模板
- 一个可复用的任务插件
- 一个把经验编码后的小模块

### 5.3.4 延迟加载为什么重要

Skill 看起来越多越强，但一个现实问题很快会出现：

> 如果所有 Skill 都预先塞进上下文，token 成本会失控。

所以延迟加载变得非常关键。

系统应该只在当前任务真正需要时，再把相关 Skill 注入进来。  
这不只是成本优化，也是上下文治理。

### 5.3.5 Skill Discovery：Agent 如何知道自己会什么

一旦 Skill 多起来，另一个问题也自然出现：

> Agent 怎么知道什么时候该用哪个 Skill？

这就需要一层发现机制。

也就是说，Skill 不是简单放在文件夹里就结束了。  
它还需要：

- 元数据
- 描述信息
- 匹配逻辑
- 加载机制

只有这样，Skill 才能真正成为系统能力的一部分，而不是作者个人的备忘录。

## 5.4 反直觉洞察：Prompt Caching 经济学

Skill 一个非常反直觉的地方在于：

很多人天然觉得“提示词越短越好”。  
但在真实系统里，这并不总是对的。

因为当 Prompt Caching 开始发挥作用时，情况会变得不一样：

- 稳定、可复用的长指令不一定更贵
- 如果它们被频繁复用，反而可能更经济
- 结构稳定的 Skill 更容易享受缓存收益

这意味着 Skill 不只是行为封装，  
它甚至会影响成本结构。

所以从产品和工程视角看，Skill 的价值至少有三层：

- 行为复用
- 质量稳定
- 成本优化

---

## 5.5 如果不写代码，怎么在脑海里理解一个 Skill

你可以想一个最典型的例子：Code Review Skill。

如果没有 Skill，Agent 每次做代码审查都可能随机发挥。  
有时候先跑 lint，有时候先看 diff，有时候只给空泛建议。

如果把这个过程封成 Skill，系统就可以稳定知道：

- 先检查什么
- 再检查什么
- 输出分成哪些部分
- 哪些问题必须优先指出

如果把这个过程再具体一点，系统实际可能会发生这样的变化：

```text
没有 Skill 时
User: 帮我 review 这个 PR
Agent: 随机发挥。有时先总结，有时先提建议，有时漏掉测试问题

加载 code-review Skill 后
User: 帮我 review 这个 PR
Agent:
1. 先运行 lint / 测试
2. 再审查行为变更
3. 最后按固定格式输出 findings
```

这个例子说明，Skill 真正改变的不是“Agent 会不会”，而是：

> 它会不会按同一套成熟方法稳定地做。

于是变化就发生了：

> Agent 不再只是“会审查代码”。  
> 它开始“按同一套成熟方法审查代码”。

这就是 Skill 真正重要的地方。

## 5.6 深入剖析：几个反直觉的洞察

### 洞察 1：Skill 本质上是“经验的序列化”

很多系统把 Skill 看成一个功能包。  
更准确地说，它是把团队经验、操作习惯、约束规则、输出格式序列化下来。

一旦这样理解，你就会明白为什么 Skill 对产品如此重要：

它让经验终于可以被系统继承。

### 洞察 2：延迟加载和预加载没有绝对正确答案

预加载的好处是简单。  
延迟加载的好处是节省上下文和成本。

真正好的系统，往往会在两者之间找平衡：

- 常用核心能力少量常驻
- 重技能按需加载

这和很多后端系统里的缓存、预热、懒加载权衡非常像。

### 洞察 3：Skill 的粒度选择会重演微服务拆分的老问题

Skill 太大：

- 上下文太重
- 职责混杂
- 复用性差

Skill 太小：

- 调度复杂
- 组合成本高
- 认知负担重

所以 Skill 粒度问题，本质上和微服务边界问题非常相似：

> 不是越细越好，也不是越大越好，而是边界要围绕真实职责来切。

### 洞察 4：从 MCP Prompts 到 Skill，说明协议层和能力层开始分化

Prompt 作为协议能力可以被标准化暴露。  
Skill 则更进一步，它把一整套任务经验封装成能力模块。

这说明系统正在从“接入能力”走向“组织能力”。

## 5.7 遗留痛点：会做事，但还会失忆

Skill 让 Agent 不再每次都从零开始重新摸索。  
但新的问题立刻出现：

> 就算它学会了方法，它记得住用户、项目和历史决策吗？

如果记不住，那 Skill 依然只能解决“怎么做”。  
却解决不了：

- 你之前偏好什么
- 这个项目已经做到了哪一步
- 上次为什么做了这个决策
- 哪些信息应该跨会话保留下来

这就是 Memory 要解决的问题。
